🛡 Nivell 7 — Unit Testing

Escut de codi: Verificació automàtica de sistemes

1. Què és el Unit Testing?

El Unit Testing és una tècnica de desenvolupament que consisteix a escriure proves automàtiques per verificar que petites unitats de codi funcionen correctament.

📌 Una unitat sol ser:

  • - Un mètode
  • - Una classe
  • - Una funció específica

🎯 Objectiu:

  • >> Detectar errors ràpidament
  • >> Garantir que el codi fa el que ha de fer
  • >> Evitar regressions
  • >> Millorar la qualitat del software

2. Què és un Test Unitari?

És un mètode que prepara dades, executa el codi i comprova el resultat.

L'Estructura AAA:

Arrange: Prepara les dades inicials.

Act: Executa el mètode que volem provar.

Assert: Comprova si el resultat és l'esperat.

3. Exemple sense testing:

public class Calculadora {
    public int Sumar(int a, int b) {
        return a + b;
    }
}

Funciona... però com sabem que sempre funcionarà després de canviar el codi?

4-5. Frameworks i Exemple real

En C# els més habituals són xUnit, NUnit i MSTest.

Exemple amb xUnit:

using Xunit;

public class CalculadoraTests {
    [Fact]
    public void Sumar_DosNumeros_RetornaResultatCorrecte() {
        // Arrange
        Calculadora calc = new Calculadora();

        // Act
        int resultat = calc.Sumar(2, 3);

        // Assert
        Assert.Equal(5, resultat);
    }
}

📌 Si el resultat no és 5 → el test falla i el sistema t'avisa.

6-7. Per què és important?

Sense tests:

- Els errors apareixen tard.

- El manteniment és difícil.

- Cada canvi pot trencar funcions antigues.

Amb tests:

- Canvis més segurs.

- Refactoring amb confiança.

- Documentació viva del sistema.

Característiques d'un bon test:

✔ Independent i Ràpid

✔ Determinista (mateix resultat)

✔ Clar i testa una sola cosa

❌ No dependre d'altres tests

❌ No usar bases de dades reals

❌ No tenir lògica complexa

8. Exemple més realista

Probem un sistema de descomptes professional:

[Fact]
public void AplicarDescompte_PercentatgeValid_RetornaPreuCorrecte() {
    SistemaDescompte sistema = new SistemaDescompte();
    decimal resultat = sistema.AplicarDescompte(100, 10);
    Assert.Equal(90, resultat);
}

[Fact]
public void AplicarDescompte_PercentatgeInvalid_LlançaExcepcio() {
    SistemaDescompte sistema = new SistemaDescompte();
    Assert.Throws<ArgumentException>(() => sistema.AplicarDescompte(100, 150));
}

9-10. Regressions i Refactoring

QUÈ ÉS UNA REGRESSIÓ?

Quan un canvi nou trenca funcionalitats que abans funcionaven. Els tests ho eviten.

REFACTORING SEGUR:

Escric tests → Escric codi → Refactoritzo. Sense tests refactoritzar és arriscat.

11-12. Cobertura i Errors Típics

La cobertura mesura quin percentatge del codi està testejat. Però recorda: 100% cobertura no garanteix 100% qualitat.

Errors típics:

  • ✖ No separar Arrange / Act / Assert
  • ✖ Escribir tests massa complexos
  • ✖ No testar casos límit
